## RAG优化：Query改写+Prompt

**文章信息**：
- **原文标题**：封神级 RAG 优化实战，query 改写 + prompt 构建，吃透面试核心考点
- **来源**：https://www.toutiao.com/article/7636023699753255424
- **作者**：AI码韵匠道（京东科技软件工程师）
- **发表时间**：2026-05-05 07:00

---

**一段话总结**：

文章围绕京东大模型二面真题"RAG 系统如何进行 query 改写及如何基于检索结果构建有效 prompt"，系统拆解了检索前优化和生成前优化两大核心环节。query 改写部分详细讲解了五种策略：指代消解、查询分解（Query Decomposition）、多角度改写（Multi-Query）、HyDE（假想文档嵌入）和 Step-back Prompting，每种都附可直接复用的 prompt 模板。prompt 构建部分从重排序（Reranker）、上下文压缩、模板设计和冲突处理四个维度展开，给出了完整的工程化 prompt 模板。文章还提供了策略组合的分级路由方案、兜底降级策略（相关性判断+混合检索），以及面试应答的标准思路框架。

---

**作者判断**：

作者认为 RAG 系统效果优化是系统性工程，query 改写和 prompt 构建虽容易被低估，但却是"提升 RAG 端到端效果最直接、成本最低的手段"。核心主张：不要无脑全上所有策略，而应"按需改写"——根据 query 类型和检索结果质量分级处理，在效果、成本和延迟之间找平衡。作者强调"兜底方案是区分玩具系统与生产系统的分水岭"。

---

**核心内容**：

**一、为什么原始 query 不能直接用于检索？**

用户的提问方式与知识库文档表述存在语义鸿沟，四种常见难检索场景：

- **口语化+模糊指代**："上次那个报错怎么解决的" vs 文档"ConnectionTimeoutException 排查步骤"
- **query 过于笼统**："介绍你们的产品" → embedding 指向模糊区域，无法精准聚焦
- **复合 query**："对比 MySQL 和 PostgreSQL 的高并发写入性能及锁机制差异" → 多子话题导致 embedding 被"平均化"
- **多轮对话指代**："它的检索阶段有哪些优化方法" → "它"无法被检索系统识别

**二、五种 query 改写策略（附 prompt 模板）**

**2.1 指代消解与上下文补全**（多轮对话刚需）

将当前 query 与对话历史结合，生成语义完整、不依赖上下文的独立 query。工程优化：先规则判断 query 是否含"它/这/那/该"等指代词，不含则跳过，降低大模型调用成本。

**2.2 查询分解（Query Decomposition）**

将复合 query 拆成多个独立子 query 分别检索。如"对比 MySQL 和 PostgreSQL 高并发写入性能及锁机制"拆为：①MySQL 高并发写入性能 ②PostgreSQL 高并发写入性能 ③MySQL 锁机制 ④PostgreSQL 锁机制。实现方式：规则分解（正则匹配"和/与/对比/以及"）用于简单场景，大模型分解用于复杂场景。

**2.3 多角度改写（Multi-Query）**

生成 3-5 个语义相同但措辞不同的 query 变体，分别检索后合并去重，覆盖知识库中不同表述风格的文档。因成本较高，建议仅在检索结果数量过少/相关性过低时触发，而非对所有 query 使用。

**2.4 HyDE（假想文档嵌入）**

不让 query 直接检索，而是让大模型先生成一篇 300-500 字假想答案文档，用其 embedding 检索。原理：假想文档与知识库文档同为"文档体"，表述风格一致，能更好匹配真实文档。同样建议在原始检索效果不佳时使用。

**2.5 Step-back Prompting**

不把 query 变得更具体，而是更抽象。如"Python 3.9 中 walrus operator 使用限制" → 抽象为"Python 赋值表达式的语法规则和使用限制"，检索更全面的背景知识。需在"抽象"和"聚焦"之间找平衡，避免过于宽泛。

**三、策略组合与分级路由方案**

核心思路：指代消解 → 判断是否复合 query（是则分解） → 初步检索 → 质量达标则直接构建 prompt，不达标则触发多角度改写或 HyDE 补充检索。

四级分级处理：

| 级别 | query 类型 | 处理策略 |
|------|-----------|---------|
| 一级 | 简单 query（单句、无指代、表述规范） | 不改写，直接检索 |
| 二级 | 有指代/省略（多轮对话） | 仅指代消解 |
| 三级 | 复合 query（多子问题） | 指代消解 + 查询分解 |
| 四级 | 检索质量不达标 | 追加多角度改写/HyDE |

工程优化技巧：①高频 query 改写+检索结果缓存复用；②高并发场景减少改写复杂度（如多角度改为 3 个变体）；③改写类任务用 GPT-3.5 级别轻量模型即可，无需 GPT-4。

**四、基于检索结果构建有效 prompt（四个环节）**

**4.1 重排序（Reranking）**

向量检索粗排 → Reranker 精排（Cohere Rerank / bge-reranker / Sentence-BERT Cross-Encoder），筛选 Top K 文档片段。关键细节：因"Lost in the Middle"现象，最相关的 2-3 个片段应放在 prompt 开头和结尾，中间放次相关内容，而非严格按分数降序排列。

**4.2 上下文压缩（Context Compression）**

对每个文档片段提取与 query 直接相关的核心信息，去掉冗余。做法：用大模型按 prompt 压缩，或使用 DistilBERT/T5-small 等压缩模型。推荐结合重排序：先重排筛选 Top K → 再逐个压缩。LangChain 提供 ContextualCompressionRetriever 组件可直接复用。

**4.3 Prompt 模板设计（三个必须要素）**

①**角色指令**：明确告诉模型只能使用检索文档中的信息，不能编造
②**引用格式规范**：要求标注信息源 [1][2]，增加可追溯性，强制模型参考文档
③**"不知道就说不知道"约束**：检索文档无相关信息时直接告知"未找到相关信息"，不强行编造

文章提供了可直接复用的完整 prompt 模板，包含五个约束项。

**4.4 文档片段冲突处理**

- 优先级排序：多版本文档标注最新版优先，官方文档优先于非官方
- 逻辑排序：存在逻辑递进关系时按"定义→组件→优化方法"顺序排列，不按相关性分数排

**五、兜底方案**

**相关性判断与降级**：Reranker 最高分低于阈值（如 0.6）时，①让模型基于自身知识回答但标注"未经验证仅供参考"，或②直接告知用户"未找到相关信息"。专业用户场景选②，普通用户场景选①。

**混合检索**：向量检索（语义匹配）+ 关键词检索 BM25（精确匹配）互补长短，通过 RRF 算法融合排序。Milvus、Pinecone 等向量库已集成。

**六、面试应答标准框架**

query 改写部分：明确目的（消除语义鸿沟）→ 分述五种策略（原理+场景+实现）→ 工程选型思路（分级路由，按需改写，平衡效果/成本/延迟）

prompt 构建部分：明确核心矛盾（token 预算有限 vs 信息量大 + Lost in the Middle）→ 分述四环节（重排序/上下文压缩/模板设计/冲突处理）→ 补充兜底方案（降级策略/混合检索）
